## 1. Database & Transaction Issues

### 1.1 Missing Transaction Rollback in Nested Operations
**Severity:** CRITICAL  
**Location:** `app/services/inventory/inventory_sync_service.py:_apply_stock_mutation_batch_mode()`

**Issue:**
The batch mode stock mutation uses nested transactions (`begin_nested()`) but the rollback logic may not properly handle all failure scenarios. If a post-transaction invariant check fails, the transaction has already been committed.

```python
with self.db.begin_nested():
    batch_result = self._execute_batch_operation(ctx)
    stock_result = self._sync_inventory_stock_to_batch(ctx)

self.db.commit()  # Already committed!

# Post-transaction check happens AFTER commit
if settings and settings.strict_consistency_check:
    is_valid, details = post_guard.verify_after_mutation(...)
    if not is_valid:
        # Too late - already committed!
        logger.critical("Post-transaction invariant violation detected")
```

**Impact:**
- Data inconsistency between batch and inventory_stock tables
- Cannot rollback after detecting invariant violations
- Silent data corruption in production

**Recommendation:**
Move invariant verification inside the nested transaction before commit.

**Severity:** HIGH  
**Location:** `app/core/database.py`

## 2. Batch Inventory System Issues

### 2.1 Batch-Stock Synchronization Race Condition
**Severity:** CRITICAL  
**Location:** `app/services/inventory/inventory_sync_service.py`

**Issue:**
The batch tracking system attempts to keep `inventory_stock` synchronized with batch totals, but the synchronization happens in separate steps:

1. Update batches
2. Commit transaction
3. Sync inventory_stock to match batch totals

Between steps 2 and 3, the data is inconsistent.

**Impact:**
- Temporary data inconsistency visible to concurrent requests
- Reports showing incorrect stock levels
- Race conditions when multiple deliveries happen simultaneously

**Recommendation:**
Use database triggers or atomic updates to ensure batch and stock are always synchronized.

### 2.2 Insufficient Stock Error Not Properly Handled
**Severity:** HIGH  
**Location:** `app/services/inventory/batch_allocation_service.py:_allocate_batches_with_cost()`

**Issue:**
When insufficient stock is detected, the savepoint rollback may not properly restore batch quantities:

```python
savepoint = self.db.begin_nested()
try:
    # ... allocate from batches
    if remaining > 0:
        raise InsufficientStockError(...)
except Exception as e:
    savepoint.rollback()  # May not restore batch.qty_on_hand
    raise e
```

The `decrement_qty_on_hand()` method modifies the batch object, but savepoint rollback may not restore the in-memory object state.

**Impact:**
- Batch quantities incorrectly decremented even when allocation fails
- Ghost inventory deductions
- Stock discrepancies accumulating over time

**Recommendation:**
Refresh batch objects after rollback or use database-level locking.

## 3. Concurrency & Race Conditions

### 3.1 Sales Order Status Update Race Condition
**Severity:** CRITICAL  
**Location:** `app/services/sales/sales_order_delivery_detail_service.py:_update_so_status()`

**Issue:**
The status update uses row-level locking but the check-then-act pattern has a race condition:

```python
# Check cancelled status early WITHOUT lock
if sales_order.status == "Cancelled":
    return

# Lock the row
locked_sales_order = (
    self.repo.db.query(SalesOrder)
    .filter(SalesOrder.sales_order_id == sales_order.sales_order_id)
    .with_for_update()
    .first()
)
```

Between the check and the lock, another transaction could change the status.

**Impact:**
- Cancelled orders being processed
- Status transitions violating business rules
- Incorrect commission calculations

**Recommendation:**
Move the cancelled check inside the locked section.

### 3.2 Purchase Order Delivery Race Condition
**Severity:** HIGH  
**Location:** `app/services/procurement/product_order_delivery_detail_service.py:create_delivery_detail()`

**Issue:**
The code locks the PO and PO detail but processes batch creation and stock updates outside the lock:

```python
# Lock PO and PO detail
self.repo.db.query(PurchaseOrder).filter(...).with_for_update().first()
purchase_order_detail = self.repo.db.query(PurchaseOrderDetail).filter(...).with_for_update().first()

# Update received quantities
purchase_order_detail.received_quantity += qty

# Update inventory stock - may create batch
self._update_inventory_stock(purchase_order_detail, qty, user_id, ...)

# Commit happens later
self.repo.db.commit()
```

If two deliveries for the same PO detail happen concurrently, both may pass validation but create duplicate batches.

**Impact:**
- Duplicate batch creation
- Over-receiving inventory
- Stock quantity errors

**Recommendation:**
Use serializable isolation level or add unique constraint on (po_detail_id, delivery_detail_id).

### 3.4 Batch Allocation SKIP LOCKED May Cause Starvation
**Severity:** MEDIUM  
**Location:** `app/repositories/inventory/batch.py` (referenced in service)

**Issue:**
The batch allocation uses `SELECT FOR UPDATE SKIP LOCKED` to avoid deadlocks:

```python
batches = self.batch_repo.lock_and_get_batches_for_allocation(
    company_id=company_id,
    product_id=product_id,
    variant_id=variant_id,
    location_id=location_id,
    order_by_fifo=order_by_fifo,
)
```

If a batch is locked by a long-running transaction, it will be skipped, potentially causing:
- Newer batches allocated before older ones (violating FIFO)
- Starvation if all batches are locked
- Inconsistent allocation order

**Impact:**
- FIFO/LIFO order not guaranteed under high concurrency
- Some orders may fail with "insufficient stock" even when stock exists
- Unfair allocation

**Recommendation:**
Use `SELECT FOR UPDATE NOWAIT` with retry logic instead of SKIP LOCKED.

## Database Schema Improvements

### Recommended Schema Changes

1. **Add Constraints**
```sql
-- Prevent negative quantities
ALTER TABLE warehouse.inventory_stock 
ADD CONSTRAINT chk_quantity_non_negative 
CHECK (quantity >= 0);

-- Enforce location for batches
ALTER TABLE inventory.batch 
ALTER COLUMN location_id SET NOT NULL;

-- Unique order numbers
ALTER TABLE sales.sales_order 
ADD CONSTRAINT uq_order_number UNIQUE (order_number);

-- Prevent negative batch quantities
ALTER TABLE inventory.batch 
ADD CONSTRAINT chk_batch_qty_non_negative 
CHECK (qty_on_hand >= 0 AND qty_received >= 0);
```

2. **Add Indexes**
```sql
-- Foreign key indexes
CREATE INDEX idx_inventory_transaction_product 
ON transaction.inventory_transaction(product_id);

CREATE INDEX idx_inventory_transaction_variant 
ON transaction.inventory_transaction(variant_id);

CREATE INDEX idx_inventory_transaction_location 
ON transaction.inventory_transaction(location_id);

-- Composite indexes for common queries
CREATE INDEX idx_batch_product_location_status 
ON inventory.batch(product_id, location_id, status) 
WHERE is_deleted = FALSE;

CREATE INDEX idx_so_detail_order_product 
ON sales.sales_order_detail(sales_order_id, product_id) 
WHERE is_deleted = FALSE;
```

3. **Add Triggers**
```sql
-- Sync inventory_stock with batch totals
CREATE OR REPLACE FUNCTION sync_inventory_stock_from_batch()
RETURNS TRIGGER AS $$
BEGIN
    UPDATE warehouse.inventory_stock
    SET quantity = (
        SELECT COALESCE(SUM(qty_on_hand), 0)
        FROM inventory.batch
        WHERE product_id = NEW.product_id
          AND variant_id <=> NEW.variant_id
          AND location_id = NEW.location_id
          AND status = 'active'
          AND is_deleted = FALSE
    )
    WHERE product_id = NEW.product_id
      AND variant_id <=> NEW.variant_id
      AND location_id = NEW.location_id;
    
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_sync_stock_after_batch_update
AFTER UPDATE OF qty_on_hand ON inventory.batch
FOR EACH ROW
EXECUTE FUNCTION sync_inventory_stock_from_batch();
```
