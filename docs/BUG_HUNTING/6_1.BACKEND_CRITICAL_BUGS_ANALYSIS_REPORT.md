## 1. Database & Transaction Issues

### 1.1 Missing Transaction Rollback in Nested Operations
**Severity:** CRITICAL  
**Location:** `app/services/inventory/inventory_sync_service.py:_apply_stock_mutation_batch_mode()`

**Analysis:** Looking at lines 680-734 in the actual code:
- The nested transaction (`begin_nested()`) is used correctly at line 680
- The commit happens at line 688 **AFTER** the nested transaction block
- Post-transaction invariant verification occurs at lines 690-705

**Status:** ⚠️ **PARTIALLY FIXED** - The code has been improved but the fundamental issue remains: the post-transaction invariant check (lines 691-705) happens AFTER the commit. If the invariant fails, it only logs a critical error but doesn't roll back. The comment at line 699 acknowledges this: "Log critical error but don't rollback since commit succeeded"

**Recommendation:** Move the invariant check inside the nested transaction before commit, or implement a compensating transaction mechanism.

## 3. Concurrency & Race Conditions

### 3.1 Sales Order Status Update Race Condition
**Severity:** CRITICAL  
**Location:** `app/services/sales/sales_order_delivery_detail_service.py:_update_so_status()`

**Analysis:** Lines 142-212 show:
- Line 151-153: Check cancelled status **BEFORE** acquiring lock
- Line 155-161: Lock with `with_for_update()`

```python

    return

    self.repo.db.query(SalesOrder)
```

**Status:** ⚠️ **PARTIALLY FIXED** - Row-level locking is implemented. However, there's still a small race window between checking cancelled status (line 152) and acquiring lock (line 158). A concurrent transaction could change status to "Cancelled" in this window. The comment "CRITICAL-020 FIX: Added row-level locking to prevent race conditions" suggests this was a fix attempt.
